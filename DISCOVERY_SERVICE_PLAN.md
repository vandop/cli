# API Discovery Service for drs

## Problem

The drs CLI currently hand-maintains 18 client methods mapping to Dremio Cloud REST endpoints. When the API changes (new endpoints, parameter changes, deprecations), the CLI drifts silently. The MCP tool descriptions are also hand-written and can desync from actual API behavior.

## Source of Truth: `dremio/js-sdk`

The [@dremio/js-sdk](https://www.npmjs.com/package/@dremio/js-sdk) TypeScript SDK is the canonical, auto-maintained representation of the Dremio API surface. It covers all three SKUs (OSS, Enterprise, Cloud) and is regenerated by the SDK team. **It is more reliable than the OpenAPI YAML specs in the monorepo**, which were never correctly exposed as public docs.

The SDK is distributed publicly as the npm package `@dremio/js-sdk`, with per-SKU entry points (`@dremio/js-sdk/oss`, `/enterprise`, `/cloud`). The SDK's TypeScript source repository is internal to Dremio and not publicly available.

### Why js-sdk over OpenAPI specs

| Factor | js-sdk | OpenAPI YAMLs |
|--------|--------|---------------|
| Maintained? | Yes, auto-regenerated | Incomplete, not all APIs exposed |
| Coverage | 97 endpoint calls across 15 resources | ~78 endpoints across scattered files |
| Structured? | TypeScript resource classes, typed request/response | Varies by file, inconsistent tagging |
| SKU separation | Explicit `oss/`, `enterprise/`, `cloud/` modules | Mixed, hard to filter |
| URL patterns | Single `getResourceConfig.ts` with all URL builders | Spread across spec files |

### Current State: Proof of Concept

`scripts/parse_jssdk.py` regex-parses the js-sdk TypeScript source to extract endpoint URLs. This proves the value but is **not production-grade** — it can't extract parameter types, request/response schemas, or survive SDK refactors. It's the pitch deck, not the solution.

Current results from the parser:
- **97 total endpoint calls** across 15 resources
- **79 unique endpoints** (deduplicated)
- **14 covered by drs** client.py
- **65 not yet in drs** (engines, AI, projects, scripts, etc.)
- **5 drs-only** (SQL-based commands with no REST equivalent)

Run it:

> **Note (internal-only prerequisite):** `scripts/parse_jssdk.py` requires the SDK's TypeScript source (`src/*.ts`), which is only available from Dremio's internal SDK repository. The public npm package `@dremio/js-sdk` ships compiled `dist/` output only and cannot be parsed by this script.

```bash
# <internal-js-sdk-repo> is Dremio's internal SDK source repository (not public).
git clone --depth 1 <internal-js-sdk-repo> /tmp/js-sdk
python scripts/parse_jssdk.py --sdk-path /tmp/js-sdk                    # generate registry
python scripts/parse_jssdk.py --sdk-path /tmp/js-sdk --compare          # compare vs drs
```

### Exploring the API today (for external users)

Until `discovery.json` exists, the recommended public ways to explore the Dremio API across all three SKUs are:

- **Official Dremio API documentation:** <https://docs.dremio.com>
- **The npm package `@dremio/js-sdk`:** install it and read the bundled `.d.ts` type declarations to see the available resources and endpoints per SKU (`@dremio/js-sdk/oss`, `/enterprise`, `/cloud`).

## Requirement: `discovery.json` Export from js-sdk

The production path requires the js-sdk team to export a structured **`discovery.json`** manifest as part of their build. The js-sdk already has all the information we need in its TypeScript types and Zod schemas — we're asking them to serialize it.

### Why the js-sdk team can do this cheaply

The data already exists in their codebase:

| What we need | Where it lives in js-sdk |
|-------------|-------------------------|
| Endpoint paths | `sonarV3Request("path")` calls in resource classes |
| HTTP methods | `method: "POST"` in request options |
| Request body schemas | Zod schemas (e.g., `jobListParamsSchema` in `listJobs.ts`) |
| Response schemas | Zod codecs (e.g., `jobDetailsCodec`, `reflectionEntityToProperties`) |
| Parameter types | TypeScript interfaces + Zod schemas |
| Deprecation markers | `@deprecated` JSDoc annotations |
| SKU separation | Directory structure (`oss/`, `enterprise/`, `cloud/`) |
| URL patterns | `getResourceConfig.ts` (already centralized) |

The implementation is a build step that walks resource classes and serializes their Zod schemas to JSON Schema. Libraries like `zod-to-json-schema` do the heavy lifting.

### What this unblocks

With `discovery.json`, the drs pipeline becomes:

```
┌──────────────────────┐
│ js-sdk build          │
│                       │
│ TypeScript + Zod      │
│    │                  │
│    ▼                  │
│ discovery.json        │  ← NEW: structured manifest, published alongside npm package
└──────────┬───────────┘
           │  npm install @dremio/js-sdk
           │  OR fetch from CDN/GitHub release
           ▼
┌──────────────────────┐
│ drs                   │
│                       │
│ registry.py           │  loads discovery.json
│    │                  │
│    ├──▶ CLI commands  │  auto-generated with typed params
│    ├──▶ MCP tools     │  auto-generated with descriptions
│    └──▶ introspect    │  `drs describe` powered by real schemas
└──────────────────────┘
```

**vs the regex parser approach (current):**

| Capability | Regex parser | discovery.json |
|-----------|-------------|----------------|
| Endpoint URLs | Yes | Yes |
| HTTP methods | Inferred (fragile) | Explicit |
| Request parameters (names, types, required) | No | Yes |
| Response schemas | No | Yes |
| Enum constraints | No | Yes (from Zod) |
| Deprecation markers | No | Yes |
| Survives SDK refactors | No | Yes |
| Auto-generate CLI `--help` | No | Yes |
| Auto-generate `--fields` options | No | Yes (from response schema) |
| Auto-generate `--dry-run` | No | Yes (from `mutating` flag) |

## Architecture (with discovery.json)

```
┌──────────────────┐
│ @dremio/js-sdk    │  (source of truth — maintained by SDK team)
│ discovery.json    │  (structured manifest — generated from Zod schemas)
└────────┬─────────┘
         │ fetched at build-time or install-time
         ▼
┌──────────────────┐
│ Registry          │  loads discovery.json + SQL_COMMANDS
│ (ApiOperation[])  │
└────────┬─────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌────────┐ ┌────────┐
│  CLI   │ │  MCP   │   Both auto-generated from the same registry
│Commands│ │ Tools  │
└────────┘ └────────┘
```

## Design: Three Layers

### Layer 1: Discovery Source

**Production (requires js-sdk team):** `discovery.json` exported from js-sdk build
**Interim (implemented):** `scripts/parse_jssdk.py` regex parser — generates `api_registry.json` with endpoint URLs only (no types)

The interim parser serves as:
1. A proof of concept showing what we'd do with structured data
2. A coverage comparison tool (drs vs full API surface)
3. A fallback if `discovery.json` isn't available yet

### Layer 2: Registry (`src/drs/registry.py`) — TO BUILD

**Input:** `discovery.json` (or `api_registry.json` as fallback)
**Output:** A structured catalog of available operations

```python
@dataclass
class ApiOperation:
    operation_id: str       # e.g., "jobs.create"
    group: str              # e.g., "query"
    subcommand: str         # e.g., "run"
    http_method: str        # GET, POST, etc.
    path_template: str      # /v0/projects/{projectId}/sql
    request_fn: str         # sonarV3Request (maps to URL builder)
    path_params: list[Param]
    query_params: list[Param]
    body_schema: dict | None
    response_schema: dict | None
    description: str        # CLI help text
    tool_description: str   # LLM-optimized MCP tool description
    deprecated: bool
    mutating: bool          # drives --dry-run behavior

class ApiRegistry:
    """Central catalog of all available API operations."""

    def __init__(self, discovery_path: Path):
        self.operations: dict[str, ApiOperation] = {}
        self._load(discovery_path)

    def get_groups(self) -> list[str]:
        """Return unique command groups (query, catalog, schema, etc.)."""

    def get_operations(self, group: str) -> list[ApiOperation]:
        """Return all operations in a group."""
```

The registry merges REST operations (from discovery.json) with SQL operations (from `SQL_COMMANDS`) into a unified command catalog.

### Layer 3: Dynamic CLI + MCP Generation

**CLI generation:**
```python
for group in registry.get_groups():
    group_app = typer.Typer(help=f"...")
    for op in registry.get_operations(group):
        cmd = build_cli_command(op)  # params from schema, --dry-run if mutating
        group_app.command(op.subcommand)(cmd)
    app.add_typer(group_app, name=group)
```

**MCP generation:**
```python
for op in registry.operations.values():
    @mcp.tool(name=f"dremio_{op.group}_{op.subcommand}", description=op.tool_description)
    async def tool_handler(**kwargs):
        return await execute(client, op, **kwargs)
```

## Hybrid Commands (SQL + REST)

Some drs commands use SQL against system tables rather than REST endpoints. These cannot be auto-generated from discovery.json. They need a separate, explicit registration:

```python
SQL_COMMANDS = {
    "jobs.list": {
        "group": "jobs",
        "subcommand": "list",
        "sql": "SELECT job_id, user_name, query_type, job_state, start_time, finish_time FROM sys.project.jobs_recent {where} ORDER BY start_time DESC LIMIT {limit}",
        "params": [
            Param("status", "string", required=False, enum=VALID_JOB_STATES),
            Param("limit", "integer", required=False, default=25),
        ],
    },
    "jobs.profile": { ... },
    "reflect.list": { ... },
    "schema.sample": { ... },
}
```

The CLI and MCP generators don't care which mechanism is used — they see the same `ApiOperation` interface.

## Grouping Rules

Maps js-sdk resources to drs CLI command groups:

```
js-sdk resource    →  drs group     Notes
───────────────────────────────────────────────────
jobs               →  query         submit, status, cancel, results
catalog            →  catalog       list, get, search
catalog (wiki/tag) →  schema        wiki, tags, lineage
catalog (grants)   →  access        per-entity grants
roles              →  access        role CRUD
users              →  access        user lookup, audit
engines            →  engine        NEW: list, start, stop, update
scripts            →  script        NEW: saved SQL scripts
ai                 →  ai            NEW: model providers, agent conversations
projects           →  project       NEW: project management
organizations      →  org           NEW: org settings
```

- [ ] GitHub Action: fetch latest `discovery.json` on js-sdk release

- [ ] Dremio Cloud serves `/api/openapi.json` at runtime
- [ ] `drs` fetches at startup, caches locally for 24h
- [ ] New endpoints appear in CLI without a drs release

## What Stays Hand-Written

Even with full discovery, some things remain manual:
- **SQL-based commands** (jobs list, job profiles, reflection list by dataset, schema sample)
- **Composite commands** that call multiple endpoints (e.g., `access audit` = get user + get roles)
- **MCP tool descriptions** for SQL-based tools (no spec to generate from)
- **Skills** (SKILL.md files) — workflow-level, not endpoint-level
- **Output formatting** — spec doesn't describe how to present data to humans
- **Input hardening** — `sanitize_input`, `sanitize_path`, validation (agent defense)

## Success Criteria

1. **`discovery.json` exists**: js-sdk build exports structured manifest with typed endpoints
2. **Zero hand-written API methods**: All REST endpoints auto-generated from discovery.json
3. **New endpoint detection**: New js-sdk release → new discovery.json → new CLI commands (via CI)
4. **Drift detection**: CI fails if drs coverage drops vs discovery.json
5. **Identical behavior**: All existing CLI commands and MCP tools produce the same output
6. **SQL commands coexist**: SQL-based commands work alongside auto-generated REST commands
